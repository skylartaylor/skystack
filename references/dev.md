# Dev Reference

Code review heuristics, architecture patterns, and things that bite you in production. Updated by `/research`.

## SQL & Data Safety

- Never string-interpolate into SQL — even `.to_i`/`.to_f` "safe" values. Use `sanitize_sql_array` or Arel.
- TOCTOU: check-then-set patterns (read a value, decide, write) need to be collapsed into an atomic `WHERE` + `update_all`.
- `update_column`/`update_columns` bypasses validations. If the field has (or should have) constraints, don't use them.
- N+1: before shipping any loop over records, verify `.includes()` is present for every association touched inside the loop. Avatar and attachment associations are the most commonly missed.
- Ruby-side `.select {}` on DB results is a code smell when a `WHERE` clause would work. Exception: intentionally avoiding a leading-wildcard `LIKE`.

## Race Conditions & Concurrency

- Read-check-write without a DB uniqueness constraint = race. Fix: add a unique index and `rescue RecordNotUnique; retry`.
- `find_or_create_by` without a unique DB index creates duplicates under concurrent load.
- Status transitions need atomic `WHERE old_status = ? UPDATE SET new_status`. Without it, two concurrent callers can both pass the check and both apply the transition.
- `html_safe` on any user-controlled string is XSS. Same for `raw()` and string interpolation into an `html_safe` context.

## Security Boundaries

- LLM output is untrusted. Emails, URLs, names generated by a model need format validation (`EMAIL_REGEXP`, `URI.parse`, `.strip`) before hitting the DB or a mailer.
- Structured LLM tool output (arrays, hashes) needs type/shape checks before any database write. Don't assume the model returned what you asked for.
- `rand()` / `Random.rand` for security-sensitive values: use `SecureRandom` instead.
- Token/secret comparisons with `==` are vulnerable to timing attacks. Use a constant-time comparison.
- Truncating data for a hash/ID (last N chars) trades entropy for convenience. SHA-256 the whole thing.

## Enum & Value Completeness

This is the most under-caught category. When a new enum value, status string, tier name, or type constant is added:

- **Trace every consumer** — don't grep and skim, actually read each file that switches on, filters by, or displays the value.
- **Check `case`/`if-elsif` chains** — does the new value fall through to the wrong default?
- **Check allowlist arrays** — search for `%w[]` literals containing sibling values and verify the new value is included everywhere it needs to be.
- **Check both ends of the stack** — a common miss is adding the value to a frontend dropdown without wiring the backend model/compute method to persist it.

The fastest way: grep for a sibling value (e.g., `"lfg"` or `"mega"`) to find all consumers, then read each one.

## Error Handling

- Log messages that say an action happened when the action was conditionally skipped. The log should say what actually occurred.
- Conditional side effects: if a record is promoted to "verified" but the URL only gets attached in one branch, the other branch creates an inconsistent record.
- Negative paths in tests should assert side effects, not just status codes. Did the record stay unchanged? Was the callback suppressed?
- `.expects(:something).never` is often missing from tests where "did not call the external service" is exactly what needs to be proven.

## Performance

- N+1 in views is the most common: associations loaded inside a loop with no preloading. Tag the loop and check the query log.
- `Array#find` inside a view loop is O(n*m). Build a hash with `index_by` before the loop.
- Inline `<style>` blocks in partials get re-parsed on every render. Extract to a stylesheet.
- Blocking the main thread with synchronous I/O or heavy computation — jobs/workers exist for a reason.
- Date-key lookups assuming "today" covers 24 hours will break for anything run early in the day (8am PT only sees midnight→8am under today's key).

## Test Gaps to Watch For

- Tests that assert type/status without checking side effects (field populated? URL attached? callback fired?).
- Happy-path tests with no corresponding negative path.
- Security enforcement features (blocking, rate limiting, auth) without an integration test that actually exercises the enforcement path end-to-end.
- String content assertions that don't check format (present vs. valid URL format).
- Time window mismatches: one feature uses hourly buckets, a related one uses daily keys — the tests pass individually but the data doesn't line up.
- Mismatched type coercion at boundaries: `{ cores: 8 }` and `{ cores: "8" }` produce different hashes. Tests that don't cross the Ruby→JSON→JS boundary won't catch it.

## Architecture Patterns

- When to add a new pattern: only when the existing pattern genuinely can't do the job. Adding a second state management library, a second HTTP client, or a second job queue because it's slightly more convenient in one case creates permanent maintenance surface.
- State management: follow what the codebase already uses. Introducing Riverpod into a Provider app or Zustand into a Redux app splits the mental model for everyone who follows.
- API design: match existing REST/GraphQL conventions. A GraphQL endpoint in an otherwise REST API needs a strong reason.
- File organization: if files of this type live in `app/services/`, put the new one there. Consistency beats having an opinion.

## Framework-Specific Gotchas

### Flutter
- Widget lifecycle: `initState` runs once; `didUpdateWidget` runs on rebuilds. Expensive setup in `build()` runs on every frame.
- `setState` in a `StatefulWidget` is fine for local, leaf-node state. Anything that needs to survive navigation or be shared: use Provider/Riverpod.
- Platform channel calls are async and can fail; always handle the error case, especially on platform-specific features.
- `dispose()` must cancel streams, timers, and animation controllers — leaking them causes hard-to-diagnose memory issues.

### SwiftUI
- `@State`: local, owned by the view. `@StateObject`: owned by the view, persists across redraws. `@ObservedObject`: owned externally, can be replaced. Using `@ObservedObject` for something the view should own causes reinit bugs.
- View identity is determined by structural position + explicit `id()`. If two views of the same type swap position, SwiftUI reuses the old state. Use `.id(someUniqueValue)` to force a fresh instance.
- Avoid heavy computation in `body` — it runs on every state change. Move it to a computed property on the model or use `task {}`.

### React
- `useEffect` cleanup: if the effect subscribes or sets a timer, the cleanup function must unsubscribe/clear. Omitting it causes stale subscriptions after unmount.
- Stale closures: a callback that closes over state from render N will still see that state when called at render N+5. Use `useRef` for values that need to be current inside a long-lived callback.
- `key` prop: using array index as key breaks reconciliation when the list reorders. Use a stable, unique ID.
- Derived state in `useState` causes double-source-of-truth bugs. Compute it from props/context instead, or use `useReducer`.

### Rails
- N+1 is the default; eager loading is opt-in. Check the query log before shipping any page that touches associations.
- Strong params: if a nested attribute isn't in `permit()`, it silently drops. Check the server log for unpermitted parameter warnings.
- Callback hell: `before_save`, `after_create`, etc. are invisible to callers and hard to test in isolation. For anything non-trivial, a service object is easier to reason about.
- `Time.now` is server local time. Use `Time.current` (respects `Time.zone`) or UTC explicitly.
